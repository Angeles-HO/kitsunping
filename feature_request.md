# Feature request template — Kitsunping

Please use this template when proposing a new feature. A clear request helps maintainers evaluate and prioritise work.

## Title

- Short, descriptive title for the feature.

## Summary

- One-paragraph summary of the feature and its purpose.

## User story / Motivation

- Who benefits from this feature and why? Describe a typical user and the problem it solves.

## Proposal

- Describe the proposed design or implementation at a high level. Include codepaths, modules, or scripts likely to be changed (e.g., `network/app/`, `policy/executor/`, `scripts/service.sh`, `net_profiles/`, `calibration/`).

## Alternatives considered

- List other approaches you considered and why they were not chosen.

## Compatibility & risks

- Backwards compatibility concerns, required permissions, and potential risks (e.g., changing kernel tunables, increased memory usage).

## Performance & resource impact

- Expected CPU, memory, storage, or battery impact. If unknown, note how to measure it.

## Testing plan

- How can maintainers/testers validate the feature? Provide steps, required hardware/emulator, and sample inputs.

## Acceptance criteria

- Concrete criteria for when the feature is considered implemented (e.g., "applies profile X within 5s of network change and logs Y").

## Implementation notes (optional)

- Suggested files to modify, pseudo-code, or links to relevant docs.

## Related issues / PRs

- Link to any existing discussions, issues, or PRs.

## Contact

- Who to contact for questions or follow-up (name, GitHub handle, or email).

# Additional notes

- If you want help drafting an initial PR for this feature, include a brief implementation sketch and I can create a starter branch/PR template for you.
