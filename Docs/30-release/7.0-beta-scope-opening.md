# 7.0-beta Scope Opening

## Purpose

This document defines the formal scope to open `7.0-beta` without mixing it with `6.30` maintenance work.

`6.30` remains the current stable maintenance line.
`7.0-beta` starts only when beta-specific work is isolated, validated, and packaged with its own narrative.

## Baseline decision

- Stable line: `6.30`
- Next milestone: `7.0-beta`
- Release interpretation: the work merged after `6.01` and before the beta opening is considered stable-line hardening, not beta scope by itself.

This includes:

- failsafe and manual rescue hardening,
- module-status cleanup,
- adaptive sampling,
- module conflict detection,
- Wi-Fi standard reporting,
- optional local ML logging and offline ONNX support,
- local fixture-based test infrastructure.

## Why 7.0-beta is still separate

The repository has not accumulated enough user-visible product change since `6.01` to justify calling the current tree an active beta by default.

The beta opening should therefore be tied to deliberate scope, not to version drift.

## In scope for opening 7.0-beta

1. Finalize canonical runtime paths and deprecation boundaries for compatibility wrappers.
2. Add second-layer tests for `network/app` and router HTTP flows using local fixtures.
3. Freeze stable `6.30` metadata so beta-only changes do not leak into maintenance packaging.
4. Document beta-only behavior, risks, and rollback expectations in one release packet.
5. Validate boot, daemon, target-engine, router push, router recommendation, and calibration interactions as one integrated runtime.

## Explicitly out of scope for 7.0-beta opening

1. Rebranding the current stable line as beta without new beta-only scope.
2. Bundling router-side private implementation into the MIT repository.
3. Expanding ML from optional local tooling into mandatory runtime inference.
4. Shipping unrelated maintenance fixes under a beta label just because they are new.

## Entry criteria

All of the following must be true before the beta tag is opened:

1. `module.prop`, `update.json`, changelog, and release notes agree on what is stable and what is beta.
2. `tools/test_all_local.sh` passes, including fixture-based runtime and router-flow tests.
3. No beta-only docs are still describing `6.30` behavior as if it were already beta.
4. Compatibility wrapper removals are either deferred or called out explicitly with rollback notes.
5. Packaging inputs are clean enough to commit without ad hoc `.gitignore` exceptions.

## Exit criteria for the opening milestone

The beta opening milestone is complete when:

1. A `7.0-beta` changelog section exists and contains only beta-scope deltas.
2. Stable `6.30` maintenance remains reproducible from the same repository metadata.
3. The router/app test layer can validate recommendation, apply, and status-push flows against local HTTP fixtures.
4. The beta release note clearly distinguishes inherited `6.30` hardening from new beta intent.

## Packaging and commit checklist

Before commit:

1. Keep `module.prop` and `update.json` on `6.30` until the beta opening commit is intentional.
2. Keep `.gitignore` focused on generated artifacts, not tracked docs or source files.
3. Ensure `testing/` fixtures are tracked and reproducible.
4. Confirm `references/` remains excluded from runtime validation and packaging decisions.
5. Run the full local runner once more and record the result in the commit message or PR description.

## Recommended opening sequence

1. Land stable-line hardening and test infrastructure under `6.30`.
2. Land beta-only docs and scope packet.
3. Flip metadata from `6.30` to `7.0-beta` in a dedicated commit.
4. Tag and publish only after the beta packet and tests are both green.
